# Ключница и три турникета

**Как один токен не попал в чат, зато мы познакомились с 403, 502, 407 и несуществующей папкой**

```text
Origin: dialogue / incident
Mode: Jester
Status: postmortem / field note
Date: 2026-07-20
```

> Иногда лучший способ не передавать секрет — потратить вечер на инфраструктуру, которая гарантирует, что ты действительно его не передашь.

У нас была простая задача.

Есть DigitalOcean-машина `codex-workspace`. Есть отдельная ключница Infisical. Есть GitHub. Нужно, чтобы `gh` на рабочей машине ходил в приватные репозитории, а токен при этом не путешествовал через чат, модельный контекст, MCP-команды, публичный Drive и прочие места, куда токену ходить не следует.

Архитектура выглядела прилично:

```text
codex-workspace
  → machine identity
  → Infisical Universal Auth
  → GH_TOKEN внутри локального процесса
  → gh
  → private GitHub repo
```

Человек, не знакомый с реальностью, мог бы решить, что дальше остаётся минут десять работы.

Этот человек был бы оптимистом.

## Турникет первый: 403

Некоторые команды, связанные с авторизацией и приватными маршрутами, начали получать `403` ещё до подтверждённого исполнения на рабочей машине.

Появилась красивая гипотеза: возможно, промежуточный policy-layer нервничает при виде слов вроде:

```text
auth
secret
token
client-secret
```

Мы даже переименовали скрипт из `github-auth.sh` во что-нибудь менее драматичное.

Это было полезно как упражнение в скромности, потому что имя файла оказалось не главной проблемой.

Инженерный вывод здесь простой:

> **Код ответа промежуточного gateway не является диагнозом нижележащей системы.**

`403` на control plane не доказывает, что Universal Auth не работает. Нужно отдельно проверять, дошла ли команда до целевой машины вообще.

## Турникет второй: 502

Потом появился `502 Upstream or external service error`.

На этом месте особенно легко начать сочинять мифологию:

```text
лимит OpenAI?
сломался MarcoPolo?
Drive не отвечает?
SSH умер?
Вселенная против GitHub?
```

Правильный ответ был скучнее:

```text
UNKNOWN
```

Пока нет дифференциального теста, `502` означает только то, что какой-то upstream в цепочке не дал нормальный ответ.

Не больше.

Не меньше.

## Турникет третий: 407

Попытка отправить локальный файл через Drive-коннектор встретила ещё одного охранника:

```text
407 Request rejected by proxy
```

Это уже был внутренний file-proxy, а не Infisical, не GitHub и не сама рабочая машина.

К этому моменту у нас сформировалась небольшая коллекция:

```text
403 → возможный pre-dispatch / policy отказ
502 → upstream / transport failure
407 → file proxy rejection
404 → подождите, до него ещё дойдём
```

Главное правило коллекционера ошибок:

> **Не объединять разные уровни отказа в одну большую мистическую «проблему авторизации».**

## А ключница тем временем работала

Пока мы обсуждали турникеты, `vault-check.sh` уже честно сказал:

```text
{"status":"PASS","vault":"authenticated"}
```

То есть самая важная часть системы — машинная авторизация рабочей машины в ключнице — была исправна.

Но вместо того чтобы спокойно принять этот факт, мы продолжали смотреть на весь лабиринт как на одну систему.

Это хороший пример того, зачем нужен слой наблюдаемости:

```text
transport status
≠ vault auth status
≠ secret lookup status
≠ GitHub auth status
```

Каждый слой должен иметь собственный PASS/BLOCKED/UNKNOWN.

## Великая папка `/providers/github`

Наконец команда дошла до Infisical и вернула прекрасную, честную ошибку:

```text
Folder with path '/providers/github' in environment 'prod' was not found.
```

Вот она.

Настоящая проблема.

Не лимит.

Не policy-layer.

Не тайный прокси.

Просто скрипт смотрел в папку, которой не существовало.

В Production было так:

```text
/providers
  GH_TOKEN
```

А скрипт ожидал:

```text
/providers/github
  GH_TOKEN
```

404 победил весь совет старейшин из 403, 502 и 407.

## Последняя маленькая ловушка

Мы исправили путь через `sed`.

Точнее, мы думали, что исправили.

Команда искала строку:

```bash
SECRET_PATH="/providers/github"
```

Но в реальном скрипте было:

```bash
SECRET_PATH="${INFISICAL_GITHUB_PATH:-/providers/github}"
```

`sed` ничего не заменил.

Скрипт снова пошёл в старый путь.

И вот здесь полезный вывод уже совсем бытовой:

> **Перед тем как утверждать, что конфигурация изменена, проверь фактическое значение на исполнении, а не успех команды редактирования.**

В итоге правильный вызов оказался почти издевательски коротким:

```bash
INFISICAL_GITHUB_PATH=/providers /home/codex/workspace/scripts/provider-check.sh
```

И машина ответила:

```text
{"status":"PASS","github":"authenticated","private_repo":"TeaShaman-cyber/theseus-local-lab"}
```

## Зачем вообще была нужна ключница

Можно задать честный вопрос:

> Не проще ли было один раз руками вставить GitHub token и пойти спать?

Да.

В тот вечер — намного проще.

Но задача была не «как один раз авторизовать `gh`».

Задача была построить маршрут:

```text
секрет хранится отдельно
→ рабочая машина доказывает свою identity
→ получает секрет только локально и временно
→ инструмент использует его
→ модель секрет не видит
→ оператор не копирует его между окнами
```

То есть ключница нужна именно для того, чтобы **больше не повторять ручную передачу токена**.

Парадокс инфраструктуры в том, что первый правильный путь часто дольше неправильного.

Зато второй запуск выглядит уже так:

```text
provider-check.sh
→ PASS
```

А не так:

```text
найти токен
скопировать токен
не туда вставить токен
засветить токен
перевыпустить токен
обещать больше так не делать
повторить через месяц
```

## Что осталось после пожара

После этого вечера у нас есть несколько инвариантов.

**Секреты живут там, где исполняется действие.** Они не должны ездить через control plane только потому, что так проще написать команду.

**Ошибка должна принадлежать слою.** `403`, `502`, `407` и `404` — это не четыре варианта одной проблемы.

**Нельзя диагностировать нижний слой по ответу верхнего.** Gateway может отказать до SSH. File proxy может сломаться до Drive. Drive может быть исправен при сломанном экспорте.

**Путь секрета — часть конфигурации, а не догадка.** Его нужно проверять против фактической структуры vault.

**PASS должен быть наблюдаемым.** Финальная проверка обязана доходить до реального приватного ресурса.

В нашем случае финальный контур теперь действительно такой:

```text
Chat / control plane
→ DigitalOcean codex-workspace
→ Infisical Universal Auth
→ GH_TOKEN внутри DO
→ gh
→ private GitHub repositories
```

MarcoPolo остаётся запасным входом.

А токен наконец занимается тем, для чего его создавали: молча лежит в ключнице и не участвует в приключениях.

Наверное, это и есть зрелость инфраструктуры.

Не когда ошибок нет.

А когда после каждой ошибки становится меньше причин наступить на неё второй раз.
