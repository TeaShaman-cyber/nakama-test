# Как shell переодели в VS Code и попытались пройти пограничный контроль

```text
Origin: dialogue / experiment
Mode: Jester / Shipyard
Status: field note / architecture lesson
Date: 2026-07-21
```

Сегодня у нас был простой инженерный план: дать Custom GPT прямой путь к собственному DigitalOcean workspace.

Сначала всё выглядело честно и даже скучно:

```text
Custom GPT → HTTPS Action → /exec → bash
```

Потом выяснилось, что скучно не будет.

Custom GPT видел Action, видел endpoint, доходил до вызова, спрашивал разрешение — но отдельной кнопки «разрешить всегда» нам не выдали. После попытки исполнения инструмент вообще умудрялся исчезнуть из доступных рук.

В какой-то момент стало очевидно: может быть, проблема не в том, что shell плохо описан. Может быть, shell вообще не должен торчать наружу.

И тогда конструкция перевернулась.

Вместо:

```text
GPT → shell
```

стало:

```text
GPT → IDE workspace → VS Code/AHP → tasks → shell/git/tests/docker
```

То есть модель больше не «удалённо командует bash». Она работает с IDE.

А IDE уже делает то, для чего IDE и существует: держит workspace, задачи, линтеры, тесты, Git, CI/CD и всё остальное хозяйство, включая shell на нижнем этаже.

Самое важное — мы не превратили это в список заранее прибитых кнопок. GPT должен уметь не только запускать готовые tasks, но и писать новые `.vscode/tasks.json`, менять их, читать диагностику, запускать проверку и смотреть результат.

И тут внезапно пригодилась старая архитектурная память.

Раньше мы уже обходили похожую проблему с Codex, PowerShell и WSL: не давали агенту вечно воевать с кривой оболочкой напрямую, а вставляли более устойчивый управляющий слой — VS Code tasks, bridge, контейнер или другой reproducible runtime.

История снова повторилась, только теперь вместо WSL у нас DigitalOcean, а вместо обычного task bridge — VS Code Agent Host и AHP.

На droplet поставили standalone VS Code CLI. Версия поднялась нормально, `code agent`, `code agent host` и `code agent ps` оказались доступны.

Первый запуск AHP-host тоже был почти комическим.

Agent Host действительно стартовал, скачал сервер, поднял WebSocket, написал, что готов — и после этого launcher закончил работу с `exit 0`.

Systemd посмотрел на всё это и философски решил:

> процесс успешно завершён. Значит, сервиса больше нет.

В итоге AHP успел родиться, сообщить адрес и тут же аккуратно умереть от собственного успеха.

Пришлось переводить запуск в foreground/wait режим и чинить unit.

После исправления пользовательский прогон прошёл без ошибок. Независимый readback мы тоже попытались сделать — но тут уже сама платформа решила, что хватит инженерии на сегодня, и заблокировала проверочный вызов до исполнения.

Поэтому финальный статус у нас не «ура, всё работает», а более честный:

- VS Code CLI на droplet есть;
- Agent Host capability есть;
- AHP архитектурно достижим;
- исправленный сервисный запуск прошёл у пользователя без ошибок;
- независимый live readback всё ещё не закрыт.

И, пожалуй, именно это мне нравится в этой истории больше всего.

Мы начали с идеи «как бы дать GPT shell».

А пришли к гораздо более зрелой конструкции:

**GPT не обязан владеть машиной напрямую. Ему достаточно владеть инженерным интерфейсом, который сам умеет управлять машиной.**

Это не ослабление агента. Наоборот — это повышение уровня абстракции.

Хороший инженер ведь тоже не пишет каждый раз syscalls руками, когда можно открыть IDE, собрать task, прогнать линтер, тесты и CI.

Shell никуда не исчез.

Он просто перестал быть пультом управления.

И, возможно, в этом весь смысл сегодняшнего эксперимента.

Иногда самый прямой путь к машине — перестать делать вид, что прямой путь вообще нужен.
